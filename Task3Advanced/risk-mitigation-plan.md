# План управления рисками

## Технические меры

| Риск | Меры снижения |
|---|---|
| Легаси блокирует миграцию | Антикоррупционные слои, Branch by Abstraction, Parallel Run, постепенный вывод DWH-логики |
| Нарушение требований безопасности | Разделение контуров, шифрование, аудит, DLP, Data Catalog с классификацией данных, выделение restricted datasets |
| Плохое качество данных | Data contracts, quality gates, schema registry, versioning, мониторинг freshness/completeness |
| Ошибки в событиях | Contract-first, backward-compatible schemas, DLQ, replay-процессы, тесты совместимости |
| Недостаточная наблюдаемость | Метрики, централизованные логи, tracing, алерты, SLO для ingestion и BI-портала |
| Утечка чувствительных данных в BI | Маскирование, row/column level security, запрет на включение медкарт и исследований в витрины |
| Рост стоимости облака | FinOps-практики, теги ресурсов, бюджеты, квоты, lifecycle policies, cold storage |

## Управленческие меры

| Риск | Меры снижения |
|---|---|
| Недостаток экспертизы | Пилоты в 1-2 доменах, архитектурный enablement, гильдии, парное проектирование, обучение |
| Сопротивление ownership-модели | Назначение Data Product Owner, прозрачные SLA, KPI по качеству и time-to-data |
| Разрыв между архитектурой и delivery | ADR, архитектурный runway, квартальные checkpoints, roadmap по доменам |
| Слишком ранний отказ от легаси | Чёткие exit criteria, phased rollout, совместное управление рисками с бизнесом |

## Приоритеты первого этапа

1. Выделить пилотные домены: `Fintech` и `Patient Flow`.
2. Ввести единый `Schema Registry` и правила совместимости.
3. Запустить `Data Catalog` и базовую модель ownership.
4. Перенести не чувствительную аналитику в self-service BI поверх Lakehouse.
5. Оставить DWH и Camel как временные мосты, а не как целевой слой.
